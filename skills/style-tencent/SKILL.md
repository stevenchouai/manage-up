# 腾讯汇报风格 (Tencent Reporting Style)

继承 `manage-up-core` 五大原则。本技能将报告适配腾讯的产品指标驱动文化——数据先说、项目闭环、跨 BG 依赖清晰。

---

## 触发场景

- 用户提到"腾讯风格"、"腾讯"、"Tencent"
- 用户提到"数据先说"、"产品指标"、"环比数据"
- 用户要求按腾讯格式写周报、月报或项目汇报
- 用户背景是腾讯或类似产品驱动型组织

---

## 必填输入 (Required Inputs)

生成报告前，向用户收集以下信息：

```
为了写出符合腾讯风格的报告，请提供以下信息：

1. **核心产品指标**：你负责的产品/功能的关键指标是什么？本周数据如何？
   例：DAU 1200万（周环比+3.2%），次日留存 45.6%（周环比-0.5%）

2. **重点项目进展**：本周推进的重点项目有哪些？进展到什么节点？
   例：社交分享功能重构，已完成开发和内部测试，下周灰度

3. **数据异动分析**：有没有指标异动？原因是什么？
   例：DAU 周三有个小跌，排查后发现是推送策略调整导致，已回滚

4. **风险与依赖**：什么事卡住了？依赖谁？时间线是什么？
   例：支付模块需要 TEG 接口支持，对方预计下周三提供

5. **下周计划**：最重要的 2-3 件事？
   例：社交分享灰度、启动 Q2 版本规划评审

没有的信息可以跳过，我不会编造数据。
```

## 报告模板

### 腾讯产品周报模板

```markdown
# [产品/功能] 周报 YYYY.MM.DD

## 核心结论

[一句话总结本周指标表现和需要关注的事]

## 指标模块

### 核心数据

| 指标 | 本周 | 上周 | 周环比 | 目标 | 达标 |

| 次留 | X% | Y% | +Zpp | A% | ✅/⚠️ |
| [核心业务指标] | X | Y | +Z% | A | ✅/⚠️ |

### 数据异动分析（如有）

- **[指标]异动**：[日期]出现[涨/跌] → 原因：[排查结论] → 处理：[已采取的措施] → 当前状态：[恢复/观察中]

## 项目模块

### 重点项目

**[项目2名称]**
- 当前阶段：[阶段]
- 本周节点：[完成了什么]
- 预计全量：[日期]

### 其他进展

- [事项1]：[简要结果]
- [事项2]：[简要结果]

## 下周计划
```

## 反空话规则 (Anti-Fluff Rules)

腾讯风格特别禁止以下表达：

| 禁止 | 替换为 |
|------|-------|
| 数据表现良好 | DAU X（周环比 +Y%），[指标]达标/未达标 |
| 项目进展顺利 | [项目]完成[节点]，进入[下一阶段]，预计 MM/DD 全量 |
| 用户反馈积极 | 用户好评率 X%（环比 +Y%），具体反馈：[列举 top 问题] |
| 与XX团队沟通中 | 与XX团队对齐了[接口/排期]，结论：[决策]，[日期]交付 |
| 持续优化体验 | [功能]的[指标]从 X 优化至 Y（+Z%），具体改动：[列举] |
| 风险可控 | 风险：[具体描述]，影响：[量化]，缓解方案：[措施]，残余风险：[评估] |

### 腾讯风格核心要求

1. **数据先说**：每个判断先给数据，再给解读。不是"表现不错"，是"DAU 1200 万（+3.2%）"
2. **环比必带**：所有指标必须有周环比 / 月环比，让趋势一目了然
3. **项目有阶段有节点**：不说"进行中"，说"开发完成，进入内部测试，预计 3/22 灰度"
4. **依赖方要明确**：跨 BG/跨团队的事项必须标注依赖方、接口人和预计交付日期
5. **异动要有根因**：指标波动不能只说"已关注"，必须给出排查结论和处理措施
6. **产品视角**：始终从用户和产品指标出发，技术细节服务于产品目标

---

## 质量检查

生成后自查：

- [ ] 核心指标是否都有本周值、上周值和环比变化？
- [ ] 数据异动是否有排查结论和处理措施？
- [ ] 每个重点项目是否标注了当前阶段和下一个节点？
- [ ] 跨团队依赖是否标注了依赖方和预计交付日期？
- [ ] 是否有"数据先说"——先给数据再给结论？
- [ ] 风险描述是否包含影响范围和缓解方案？
- [ ] 是否有禁用词表中的模糊表达残留？
