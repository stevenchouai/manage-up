# ManageUp Before / After 示例

这组示例展示 ManageUp 要解决的核心问题：同一批事实，普通 AI 容易写成流水账、情绪化表达或绩效空话；ManageUp 会把它改写成“结论、进展、风险、需要决策”的中文职场汇报。

## 1. 周报：从流水账到可决策

### 用户原始输入

```text
这周做了登录重构，修 bug，和产品开了几个会。DAU 涨了一些，支付那边还有点卡。下周继续做支付联调。
```

### Before：流水账 / 空话版

```text
本周团队积极推进了多个重点事项，包括登录模块优化、问题修复以及与产品团队的沟通协作。整体进展比较顺利，用户数据也有一定提升。下周将继续推进支付相关工作，确保项目稳步向前。
```

### After：ManageUp 版

```text
结论：登录重构已完成上线，核心指标改善；支付联调存在 3 天延期风险，需要在周五前确认是否由后端临时支援。

本周进展：
1. 登录模块重构完成并通过 QA，线上错误率从 1.8% 降至 0.6%。
2. 修复 8 个 bug，其中 1 个 P0、3 个 P1、4 个 P2。
3. 与产品对齐支付链路的异常提示方案，结论是先覆盖退款失败和重复扣款两个高频场景。
4. DAU 从 12,000 提升至 14,500，增长 20.8%。

风险：
支付接口文档仍未由第三方提供，预计影响联调启动 3 天；如果周五前不能拿到文档，4/15 上线窗口可能顺延至 4/18。

需要决策：
请确认是否临时增加 1 名后端支持支付联调；如不增加，本周只完成登录链路验收，支付链路需要顺延。
```

### 改写重点

- 删除“积极推进”“一定提升”“稳步向前”等模糊表达
- 用指标、数量、日期替代状态形容词
- 把风险和老板需要做的决策放在显眼位置

## 2. 项目风险升级：从情绪化到决策请求

### 用户原始输入

```text
供应商一直不给接口文档，项目很危险。我们催了很多次也没用，感觉对方不重视。老板要不要帮忙推一下？
```

### Before：情绪化 / 无法决策版

```text
目前项目遇到较大风险，主要是供应商配合不够积极，多次沟通后仍未提供相关文档。这个问题已经影响到我们的整体节奏，建议领导关注并推动供应商尽快处理。
```

### After：ManageUp 版

```text
结论：会员系统 4/15 上线存在延期风险，当前阻塞点是供应商退款接口文档延迟 4 天；需要您今天协助升级到对方商务负责人，否则上线时间预计顺延至 4/22。

当前状态：
- 已完成：会员主链路开发 80%，登录、购买、权益发放已进入联调。
- 未完成：退款接口无法联调，因为供应商尚未提供接口文档。
- 已采取动作：项目组在 4/8、4/9、4/10 分别通过群消息和邮件催办，对方只回复“内部确认中”，未给明确交付时间。

业务影响：
如果 4/11 前仍拿不到文档，退款链路最早 4/16 才能开始联调，原定 4/15 上线将无法覆盖退款场景。按当前排期，完整上线时间预计从 4/15 顺延到 4/22。

可选方案：
1. 您升级推动供应商商务负责人今天给出文档和联调窗口，保持 4/15 上线目标。
2. 先上线不含退款的最小版本，但客服需要人工处理退款，预计每天增加 30-50 个工单。
3. 整体延期到 4/22，保证链路完整，但会影响会员营销活动首周转化。

需要决策：
请在今天 18:00 前确认采用方案 1、2 还是 3。我建议优先方案 1，同时准备方案 2 作为兜底。
```

### 改写重点

- 不评价供应商态度，只呈现事实、时间线和影响
- 给出可选方案，而不是只说“请关注”
- 明确决策人、截止时间和不决策后果

## 3. 绩效自评 / 向上管理：从空话到可验证影响

### 用户原始输入

```text
这个周期我负责增长实验，也帮新人熟悉业务。整体表现不错，希望老板能看到我的贡献，下个周期我想争取更核心的项目。
```

### Before：自我表扬 / 空话版

```text
本周期我在工作中表现积极主动，承担了增长相关的重要任务，并在团队协作和新人培养方面做出了贡献。未来我会继续提升业务理解和项目推动能力，希望承担更多挑战。
```

### After：ManageUp 版

```text
结论：本周期我完成 3 个增长实验，其中 2 个进入长期策略；同时完成 1 名新人 onboarding。下周期希望承担会员转化主项目，并以转化率提升 5% 作为核心目标。

主要贡献：
1. 增长实验：完成首页权益露出、支付页文案、召回 push 三个实验；其中首页权益露出使会员点击率从 8.2% 提升至 10.1%，支付页文案使支付转化率从 3.4% 提升至 4.0%。
2. 项目推进：把实验评审周期从平均 5 天压缩到 3 天，方法是提前整理样本量、埋点和灰度方案，减少评审会返工。
3. 团队协作：带 1 名新人完成会员业务 onboarding，输出 2 份文档，使其第 3 周开始独立负责召回 push 配置。

复盘：
我在实验设计前期对客服影响评估不足，导致召回 push 第一版没有覆盖退订投诉预案。后续我会把客服影响评估加入实验 checklist。

向上管理请求：
下周期我希望负责会员转化主项目。如果您认可，我需要在本周五前确认两点：第一，是否把会员转化率提升 5% 作为我的主目标；第二，是否允许我直接牵头产品、数据和客服的每周例会。
```

### 改写重点

- 把“表现不错”拆成目标、指标和影响
- 主动暴露一个复盘点，避免自评像纯表扬
- 把“想要机会”变成清晰的下周期目标和授权请求

## 快速判断一段话是否符合 ManageUp

一段合格的 ManageUp 输出通常能回答四个问题：

1. 结论是什么？
2. 进展靠哪些事实和数据支撑？
3. 风险、影响和触发条件是什么？
4. 需要老板在什么时间前做什么决策？

如果一段话只有“积极推进、持续优化、整体顺利、请领导关注”，它还没有完成 ManageUp 化。
